Licensing Open Source Code for Use in Commercial Products

Michael Pobega

CS307 Software Engineering

 Licensing outside code for use in a commercial product can be a tricky endeavor; it is going to be either expensive to license, or timely to code from scratch. Selling source code to other companies isn’t unheard of (for example, the Quake 3 and Unreal game engines used to be sold to alleviate the time spent programming the actual physics engine; all that would have to be changed is a few variables to get it working any way a company wants), but outside of a few mainstream SDKs and libraries the practice isn’t as common as it used to be.

 Making use of already-made code in a project is a smart idea, especially if the team is on a tight budget or short schedule; why reinvent the wheel when it’s probably been invented multiple times already? Libraries were created for just this purpose, the reuse of code. Finding a suitable library can sometimes be troublesome, but usually it works in favor of the development team and can increase the throughput by a large amount. It also decreases the time it takes to do testing and spiking, since the tools being used should be completed already (no team should license incomplete code.) 

 Proprietary code is expensive, and for a low budget maybe not suitable but using commercial code without purchasing a license can lead to lawsuits. Recently, Microsoft was sued by Alcatel-Lucent[0] for using the mp3 library and format without paying royalties/licensing the library. Microsoft could have avoided this issue by adopting open source formats instead, such as Ogg Vorbis[1] Microsoft could have avoided the lawsuit and distributed music without such a huge debt to pay.

Proprietary licensing is expensive though, and may even cut into a projects budget by too much to be affordable. Licensing code is usually not an option for small budget projects, as well. Assuming an alternative exists, free software could be used in place of licensed/proprietary libraries or code.

 Using open source tools and software is a great way to decrease development time, leaving more time available for debugging and ensuring your product works, and saving money as it is generally free. Open source software (from now on referred to as “OSS”) has reinvented most libraries and programs, and a quick look through Sourceforge ( http://sourceforge.net/ ) shows that there are literally hundreds and thousands of OSS projects in constant development. 

OSS is very community focused -- a programmer will write a program for the better of the community, get feedback from other members and maybe even some coding help, all while releasing the code to a public repository for viewing and using. For a bit of history, before this kind of community existed a man named Richard Stallman wanted a free operating system and program suite for personal use by anyone, so he began the GNU operating system project (which ended up merging with the Linux kernel project and became the robust Linux OS seen today), and the Free Software Foundation (FSF). To protect the rights of the programmers Stallman instituted the GNU General Public License, which ends up being more restrictive then it does freeing.

Another commonly used license is the 3-clause BSD license[2], which has next to no restrictions with the exception of having to include the upstream author in the credits for the program. In comparison to the BSD license, the GPL has far more restrictions is much more alike to a legal document than a freedom-promoting license.

 Implementing any BSD-licensed code into a proprietary project is trivial; as long as you give the original author credit you can use it however you want, without even having to make changes to the original code available.

GPL on the other hand is more restrictive in that sense; any GPL code that is statically linked into a product must have source available (for both the GPL code and the project), thus forcing projects to become GPL’d when they use GPL. GPL is commonly referred to as a ‘viral’ license, as it requires any code that uses GPL code to become GPL itself[figure 4.1]. Obviously this is counter-intuitive for companies, since open sourcing a commercial product (unless part of the original planning) is generally not a good idea.

Static linking

figure 4.1

There are ways around the GPL’s viral nature; one way is to ask for an LGPL licensed version of the product, which places copyleft restrictions on the code itself but not any programs that make use of the code, though changes to the code would still need to be made publicly available.

In the case of GPL code though, there are a few ways around the viral nature without needing the author to re-license the code. One very popular method is the “GPL Linking Exception”[figure 4.2][3], which modifies the GPL by dynamically linking the code rather than statically linking. This gets around the restriction of having to license the product’s code as GPL, and allows for use of the GPL code.

Dynamic linking

figure 4.2

Another method that gets a lot of use is the plugin system method, in which you make the product in a way that doesn’t rely on the GPL’d library, then add in support for the library as a plugin (for example, a WYSIWYG editor in a forum software, using a GPL WYSIWYG editor.) Of course this method doesn’t work if the library is integral to the inner workings of the program, but in lieu of this method the GPL linking exception can be used.

 BSD code requires no tricks to use, as the license itself is only one step above unlicensed/free domain. Many projects are based on BSD code, Mac OSX being the primary example. Mac OSX is based on Darwin BSD code, which is based on the original Unix BSD. Under the flashy graphics and gimmicks of OSX lies a BSD-licensed system at heart, and due to the nature of BSD Apple didn’t have to jump through hoops to use the code.

 In converse to that, using GPL code is generally much trickier. TiVo incorporated GPL code in it’s system, but used hardware limitations to make the code inaccessible by the users, which essentially makes it proprietary. In response Richard Stallman coined the term “Tivoization”[4] from this, and rewrote the GPL to prevent loopholes like this from happening in the future (which were eventually incorporated to the GPLv3).
